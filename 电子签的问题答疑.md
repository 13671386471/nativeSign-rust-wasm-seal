# 1、我是一名程序员，现在要开发一套电子签名系统，想要了解电子签名系统的流程；特别是电子盖章这一步骤；请使用合适的方式向我介绍
## 电子签名 & 电子签章系统完整业务 + 技术流程（程序员开发版）
## 一、整体系统核心定位
依据《电子签名法》，可靠电子签名才具备与手写签名、实体公章同等法律效力，核心满足四要素：

1、签名人身份唯一可识别；

2、签名动作由签名人专有控制；

3、签署后内容、签章无法篡改；

4、签署行为可追溯、可验签。

电子盖章（电子签章）是电子签名的可视化载体：底层是数字签名密码算法，上层渲染公章图片（企业章、法人章、合同骑缝章、个人手写签）。

## 二、全流程总览（从发起文件→盖章→存证验签）
### 阶段 1：用户实名认证（前置，盖章必备）
#### 1. 个人用户实名

要素：身份证 OCR 识别、人脸活体核验、手机号三要素比对；

输出：唯一实名身份标识，对接 CA 机构申请个人数字证书（私钥 + 公钥证书）。

#### 2. 企业用户实名（电子盖章主体，重点）
1、资料采集：营业执照、法人身份证、对公账户、授权经办人信息；

2、核验：工商数据比对、法人人脸核验、对公打款验证；

3、签发企业数字证书：企业证书绑定统一社会信用代码，企业私钥是盖章的核心密钥；
```
法律关键点：没有合法 CA 颁发数字证书的电子章，不具备强司法效力。
```
#### 密钥存储方案（开发选型）
表格
| 方案 | 适用场景 | 安全等级 |
| --- | --- | --- |
| 云密钥托管（第三方 CA / 存证平台） | SaaS 电子签、中小企业快速接入 | 高，密钥隔离，无需自研加密硬件 |
| 本地硬件加密机 / USB-KEY | 政企、金融高安全场景，自有系统 | 极高，私钥不出硬件，符合等保三级 |
| 软件密钥（本地加密存储） | 内部简易审批、对内文件，不对外合同 | 低，不推荐对外商用签章 |
### 阶段 2：文件预处理（盖章前统一处理）
支持文件：PDF（行业标准，唯一适合电子签章格式）、Word 需转 PDF、图片需合成 PDF。
流程：

1、上传文件，服务端校验文件格式、病毒查杀；

2、PDF 标准化处理：扁平化图层、提取文档唯一原文哈希值（SHA256）；

3、支持多签署位置标记：前端拖拽坐标，记录页码、x/y 偏移、签章尺寸、骑缝标记；

4、区分签署类型：个人手写签名、企业公章、财务章、法人章、骑缝章、骑缝骑章。
### 阶段 3：电子盖章核心流程（技术拆解，开发重点）
#### 3.1 区分两种签章模式
1、可视化图片签章（用户肉眼看到的公章图片）
只是图层图片，无法律效力，必须绑定底层数字签名；

2、密码学数字签章（法律生效核心）
使用证书私钥对文件哈希加密生成签名值，嵌入 PDF 标准签名域。

#### 3.2 分步盖章技术流程（标准 PDF PAdES 签章，国内通用）
#### 步骤 1：锁定签署人身份与权限校验
1、判断当前操作账号是否持有企业盖章权限（后台配置公章授权：哪些人可盖、哪些合同类型可用）；

2、二次鉴权：盖章前强制校验（短信验证码 / 人脸 / USB-key PIN 码），满足 “签名人专有控制” 法律要求；

3、加载对应公章资源：企业备案公章透明底 PNG 图片（提前上传、备案存档）。

#### 步骤 2：计算文件原文哈希（防篡改根基）
服务端读取待签署 PDF 全部内容，使用 SHA-256 生成固定摘要 Hash；
```
逻辑：文件哪怕多一个空格、改一个字，Hash 完全变化，验签时可直接发现篡改。
```
#### 步骤 3：使用证书私钥对 Hash 进行加密（生成数字签名）
1、获取该企业 / 个人对应的签名私钥；

2、采用国密 SM2（国内合规强制推荐）/RSA 算法加密文件 Hash，生成原始签名值；

3、组装签名元数据（司法追溯关键）：

    3.1 签署主体信息（姓名 / 企业统一社会信用代码）

    3.2 CA 数字证书序列号、颁发机构

    3.3 精确签署时间戳（权威时间服务 TSA，防篡改签署时间）

    3.4 签章位置坐标、印章类型、设备标识、IP 地址、操作人

    3.5 合同编号、业务流水号
#### 步骤 4：PDF 图层合成可视化电子公章
  前端 / 服务端 PDF 渲染层操作：

    1、根据预设坐标，在 PDF 指定页码图层叠加透明公章 PNG；

    2、骑缝章特殊逻辑：自动拆分多页 PDF，每页边缘截取章图片段，拼接形成完整骑缝效果；

    3、图层层级控制：公章置于文字上层，不遮挡关键合同内容；

    4、支持设置签章外观：包含企业名称、证书编号、签署时间小字（标准合规样式）。
#### 步骤 5：将数字签名、元数据嵌入 PDF 标准签名域
PDF 规范预留 /Sig 签名字典，把加密后的签名值、时间戳、证书链全部写入文件内部，形成带签章的合规 PDF；
关键点：

    1、标准 PAdES-B-LT 格式，法院、公证处均可直接解析验签；
    2、不只是贴图片，签名数据和文件绑定，删除印章图片也无法清除底层数字签名。
#### 步骤 6：盖章后固化文件，禁止二次无授权修改
    PDF 添加签名锁：签署区域及全文锁定，普通编辑器无法删除、移动、覆盖电子章；如需修改必须发起 “撤销签署 + 重新盖章” 流程。
### 阶段 4：签署完成后置流程
1、文件存证

    签署完成 PDF、数字签名原文、操作日志、实名记录同步至第三方存证平台 / 联盟链；
    链存优势：存证记录不可删除，司法出证快速。

2、回执生成

    生成签署回执单：包含文件哈希、证书信息、签署时间、验签入口，发送给甲乙双方。

3、消息通知

    短信 / 站内信 / 邮件通知对方查看已盖章合同。
4、归档管理

    按企业、合同类型、时间分片存储原始签署文件，留存至少 5 年（合规要求）。
### 阶段 5：验签流程（盖章后的校验功能）
用户 / 法院上传已盖章 PDF，系统自动校验：

    1、提取 PDF 内部数字签名值；
    2、使用证书公钥解密签名，还原原始文件哈希；
    3、重新计算当前 PDF 文件哈希，二者比对：一致 = 未篡改，不一致 = 文件被修改；
    4、校验时间戳有效性、数字证书是否过期 / 吊销；
    5、输出验签报告：签章主体、是否有效、是否篡改、证书状态。
## 三、特殊场景：骑缝电子盖章专属逻辑（开发难点）
    1、自动获取 PDF 总页数 N；
    2、完整公章图片横向均分 N 份；
    3、每一页侧边嵌入对应分段印章图片；
    4、底层仍只生成一份全局数字签名（整份文件统一哈希，不需要每页单独签名）；
    5、验签时自动校验全部骑缝片段完整性，若某一页被删除，哈希不匹配直接判定篡改。

## 五、合规 & 开发避坑重点（商用系统必须遵守）
    1、算法强制：国内商用系统优先国密 SM2/SM3，仅 RSA 无法满足等保、司法认可；
    2、时间戳不可自建：必须对接国家认可 TSA 时间服务，本地系统时间可篡改，无法律效力；
    3、私钥安全红线：企业盖章私钥不能明文存在数据库、前端，禁止前端完成签名运算（前端只渲染图片，签名加密逻辑放服务端 / 加密机）；
    4、公章备案：企业电子章图片建议同步至 CA / 存证平台备案，避免伪造章争议；
    5、禁止仅图片签章：只贴公章图片、无数字签名的系统，发生纠纷法院不予认可；
    6、多会签场景：每一方盖章都会生成独立签名域，多方签名互不覆盖，可分别验签每个签署人。

# 2、无论是个人还是企业 数字证书 就是（私钥 + 公钥证书）是这样理解的吗？
结论先行
不完全准确，分两层纠正你的理解：
1. 数字证书本身 ≠ 私钥 + 公钥证书
数字证书（CA 证书）只包含：公钥 + 身份信息 + CA 签名
证书里永远不会存放私钥，私钥是独立、保密、仅持有者保管的隐私数据。
标准 X.509 数字证书内容（公钥载体）：
版本、序列号、有效期
持有者身份（个人姓名 / 企业统一社会信用代码）
持有者公钥（明文公开）
CA 机构信息、CA 自身签名、扩展字段（用途、CRL 吊销地址等）
2. 一套完整签名身份载体 = 私钥 + 公钥数字证书
你这句话正确修正版：
个人 / 企业用于电子签名、电子盖章的完整密码对套装 = 私钥（保密） + CA 颁发的公钥证书（公开）
拆分讲清楚（贴合电子签章系统业务）
一、公私钥对（原生密码对，无法律身份）
通过 SM2/RSA 算法生成一对数学关联密钥：
私钥（Private Key）
绝密，仅签名人持有；
用来做签名（盖章、签署文件）；
泄露 = 公章被盗用，任何人可用它伪造签章；
CA 平台 / 加密机托管，不会下发给前端 / 客户端。
公钥（Public Key）
可对外公开；
用来验签，验证签名是不是对应私钥生成；
单独的公钥没有身份含义，别人无法知道这个公钥属于谁。
二、数字证书 = 给公钥绑定合法身份（法律核心）
单纯公私钥对只能做加密验签，但没有法律效力；
CA 机构核验实名身份后，用 CA 根私钥对你的「公钥 + 身份信息」做签名，打包生成 X.509 证书：
作用：第三方可信背书 ——这个公钥确实属于该个人 / 企业。
证书只存公钥，不含私钥；
证书可以公开分发、嵌入 PDF 签章、对外提供给对方验签；
证书过期 / 吊销后，对应的签名会判定无效。
三、日常开发里的两种文件形态（你会实际对接）
1. 签章方本地 / CA 托管整套材料（签署用）
私钥.key + 公钥证书.cer + CA 根证书链
盖章签名时：只用私钥运算；
嵌入 PDF 时：把公钥证书、证书链塞进签名域，供别人验签。
2. 验签方只需要：公钥证书
不需要私钥，提取证书内公钥即可解密校验文件签名。
四、结合电子盖章场景举例
企业实名后，CA 生成一对 SM2 公私钥；
CA 用自己根证书签名企业公钥 + 企业名称 / 信用代码，生成企业数字证书（仅含公钥）；
私钥托管在 CA 密钥服务器，企业盖章时，服务端传文件哈希到 CA，用该私钥完成 SM2 签名；
盖章 PDF 内嵌这份企业数字证书；
法院 / 客户验签：读取 PDF 内证书得到企业公钥，校验签名，同时通过证书确认签署主体真实身份。
五、你容易混淆的两个误区
误区 1：数字证书包含私钥
错。标准 X.509 证书无私钥。
只有 PFX/P12 加密包是特例：把证书 + 私钥打包加密，设置访问密码，用于本地导出备份，证书本身结构依然分开公私钥。
误区 2：有公私钥就能做合法电子签章
错。无 CA 背书证书的裸公私钥，无法证明身份，司法不认可；
可靠电子签名四要素里，身份可信依赖 CA 数字证书。
极简总结
数字证书 ≠ 私钥 + 证书；证书仅承载公钥 + 可信身份；
完整可用于合法电子盖章的身份套件 = 独立保密私钥 + CA 签发的公钥数字证书；
私钥负责签名（盖章），证书（公钥）负责验签 + 证明签署人身份。


# 3、个人 / 企业的私钥是什么时候生成的？
一、核心结论
个人 / 企业 SM2/RSA 公私钥对（含私钥）在 CA 机构完成实名核验通过后、颁发数字证书之前生成，生成主体分两种主流商用模式：
CA 服务器端生成（电子签章 SaaS 主流，90% 商用系统）
客户端本地生成（USB-KEY 硬件、高安全政企场景）
私钥绝对不能先预生成、不能用户自己随便生成后上传 CA，有合规与安全风险。
二、两种生成场景完整时序（贴合你的电子盖章系统）
模式 1：CA 云端托管密钥（SaaS 电子签通用，推荐中小企业）
时间线节点：
用户提交实名材料（个人身份证人脸 / 企业执照 + 法人核验）；
CA 后台人工 / 自动完成实名审核，审核通过；
CA 密钥服务器内部调用 GMSSL 密码库，生成一组全新 SM2 公私钥对；
私钥：加密存入加密机 / 密钥托管库，永不对外输出；
公钥：提取出来，绑定用户实名信息；
CA 用自身根私钥对「公钥 + 身份信息」签名，生成 X.509 数字证书；
把数字证书返回给业务系统用于签章，私钥全程留在 CA 内部；
关键特点：
私钥生成时机：实名核验成功后、出证书前；
开发者（你的系统）全程看不到、接触不到私钥；
盖章逻辑：你只传文件哈希给 CA，由 CA 内部用私钥做 SM2 签名。
模式 2：本地硬件生成（USB-KEY / 加密 U 盾，金融、政务等高安全场景）
时间线节点：
用户完成线上实名初审，线下领取 USB-KEY 硬件；
用户插入 KEY，硬件芯片内部独立生成公私钥对；
私钥永久锁在硬件安全芯片内，无法导出、无法复制；
硬件仅把公钥上传至 CA；
CA 核验身份后，针对上传的公钥签发数字证书；
证书下载写入 USB-KEY；
关键特点：
私钥生成时机：用户激活硬件 U 盾时；
私钥永远不出硬件，CA、业务系统都拿不到；
盖章：业务服务转发哈希到客户端硬件，硬件内部完成签名。
三、纠正两个常见开发误区
误区 1：业务系统（Node/Java 后端）自己生成公私钥再发给 CA 办证
不推荐，且不合规：
业务服务器内存会短暂出现明文私钥，等保、电子签名安全规范不允许；
一旦服务器被入侵，批量泄露大量企业私钥，法律责任极大；
正规第三方 CA 不支持 “外部公钥换证” 模式，只支持 CA 内部生成密钥。
误区 2：用户前端浏览器生成密钥对
极度危险：
JS 内存可被窃取，私钥明文留存浏览器内存，无任何安全防护，司法不认可这类签名。
四、补充配套生命周期节点
证书续期：证书到期前，重新走一遍实名复核 → 生成新公私钥 → 新发证书；旧私钥作废；
证书吊销：企业注销、人员离职，CA 标记证书吊销，原有私钥永久失效；
密钥更换：企业公章变更、经办人变更，会重新生成一组全新私钥 + 新证书。
五、极简一句话总结
私钥统一在实名身份核验全部通过后、CA 签发数字证书的前一步生成；
分两种位置：要么在 CA 加密服务器内部生成托管，要么在用户 USB 硬件芯片本地生成，业务应用层不参与私钥生成。


# 4、个人 / 企业 SM2/RSA 公私钥对（含私钥）在 CA 机构完成实名核验通过后、颁发数字证书之前生成   把这个流程拆开详细讲解下

详细拆解：实名核验通过 → 生成 SM2/RSA 公私钥对 → CA 签发数字证书完整流程
适用场景：市面上绝大多数电子签章 SaaS、第三方 CA 云端托管密钥模式（私钥生成在 CA 加密服务器，业务系统不触碰私钥）
前置前提
用户（个人 / 企业）先在业务系统提交实名资料，还未走到密钥、证书环节。
整体阶段拆分
用户提交实名材料并发起核验申请
CA 侧全维度实名核验（个人 / 企业两套核验逻辑）
核验结果判定：驳回 / 通过
核验通过：CA 加密机独立生成 SM2/RSA 公私钥对（核心：私钥诞生节点）
提取公钥，绑定实名身份信息，组装证书申请载体
CA 根证书私钥签名，生成标准 X.509 数字证书
证书下发业务系统，私钥加密持久化托管在 CA 密钥机房
业务系统绑定证书与用户，后续可用于电子签名 / 电子盖章
阶段 1：用户提交实名材料，业务系统转发至 CA
1.1 个人用户材料
身份证正反面 OCR、活体人脸核验视频、手机号三要素实名（姓名 + 身份证号 + 手机号一致）
1.2 企业用户材料
营业执照、统一社会信用代码、法人身份证、法人人脸核验、对公账户小额打款验证、经办人授权书
1.3 交互逻辑
前端收集资料，上传到自研业务服务；
业务服务将脱敏后的实名数据调用 CA 开放 API，发起实名核验工单；
CA 系统接收工单，分配审核队列。
阶段 2：CA 执行多层实名核验（区分个人 / 企业）
2.1 个人核验链路
对接公安人口数据库，校验身份证真伪、有效期；
人脸比对：活体防翻拍、防照片攻击，人脸与身份证头像匹配度校验；
运营商三要素校验，确认手机号归属人；
内部风控：黑名单、重复实名、异常操作 IP 拦截。
2.2 企业核验链路
对接市场监督管理局工商库，核验营业执照真实有效、企业存续状态；
法人信息比对：法人身份证真实、人脸核验通过；
对公账户打款校验：CA 向企业对公账户打入随机小额金额，用户回填金额完成账户所有权验证；
校验经办人授权文件，确认经办人具备代办签章证书权限；
企业风控：经营异常、失信企业、黑名单直接拦截。
阶段 3：核验结果分流
核验不通过：CA 返回驳回原因（人脸不匹配、执照过期、打款金额错误等），流程终止，不生成任何密钥、不签发证书；
核验全部通过：工单状态变更为「已实名，待生成密钥」，进入密钥生成环节（你提问的核心节点）。
阶段 4：CA 加密机生成 SM2/RSA 公私钥对（私钥生成关键步骤）
4.1 环境隔离说明
密钥不会在普通应用服务器生成，仅在国密合规硬件加密机 / 密码机内部运算：
加密机物理隔离、独立机房、防火墙隔离；
具备防拆解、内存自动清零、抗侧信道攻击硬件防护；
业务系统、CA 业务后台均无法读取加密机内部私钥明文。
4.2 分步生成逻辑
CA 业务服务向加密机下发指令：为当前实名工单生成一组 SM2 公私钥（国内电子签章强制优先 SM2，RSA 仅兼容老旧系统）；
硬件密码芯片通过真随机数发生器生成安全随机种子；
基于国密 SM2 椭圆曲线算法，计算生成一对强关联密钥：
私钥（d）：绝密参数，全程驻留在加密机安全内存；
公钥（P）：由私钥推导得出，可安全导出。
加密机自动执行两条存储逻辑：
① 私钥：采用国密 SM4 加密后，存入加密机内置安全存储，禁止导出、禁止明文输出；分配唯一密钥 ID，与当前实名主体绑定；
② 公钥：明文导出，返回给 CA 业务服务。
重点回答你的核心疑问：私钥何时诞生？
在实名 100% 核验通过之后，数字证书签发之前，由 CA 硬件加密机内部生成；
未完成实名绝对不会提前创建任何密钥，避免空身份无效密钥占用资源、产生安全风险。
阶段 5：组装证书请求信息，绑定公钥 + 实名身份
现在仅有公钥，无法证明公钥归属谁，需要把身份和公钥打包，等待 CA 签发证书：
组装 X.509 证书核心字段：
主体身份信息
个人：姓名、身份证号
企业：企业名称、统一社会信用代码、法人名称
本次加密机生成的公钥（SM2 公钥）
证书基础配置：证书有效期（一般 1 年 / 3 年）、证书用途（电子签名、电子签章专用）、密钥用法、CRL 证书吊销地址
工单唯一编号、实名核验流水号（用于司法溯源）
以上整体构成待签名的证书明文数据包。
阶段 6：CA 根私钥签名，生成正式数字证书
CA 根密钥同样托管在顶级加密机中，仅授权签名接口可调用；
使用 CA 机构自身根证书对应的根私钥，对上面「身份 + 公钥」数据包进行 SM2 数字签名；
将 CA 签名、CA 机构信息、根证书链一同封装，生成标准 X.509 格式公钥证书（cer/crt 格式）；
关键点：这份证书里只包含公钥，不存在私钥。
阶段 7：证书下发、私钥永久托管
CA 将生成好的公钥数字证书通过 API 返回给你的电子签章业务系统；
业务系统将证书 ID、证书文本与用户 / 企业账号关联存储；
私钥全程留在 CA 加密机：
业务系统永远拿不到私钥；
后续电子盖章时，业务服务仅上传 PDF 文件 SM3 哈希 + 证书 ID 到 CA；
CA 内部根据证书 ID 找到对应私钥，在加密机内完成 SM2 签名运算，只返回签名结果，不暴露私钥。
阶段 8：证书交付业务系统，开启签章能力
业务后台展示当前企业 / 个人有效数字证书；
支持上传公章图片，将印章模板与该证书绑定；
可正常发起合同上传、标记签章位置、执行电子盖章流程。
配套补充关键规则 & 开发避坑
1. 为什么不能业务系统自己生成公私钥再传给 CA 办证？
业务服务器内存会短暂存在明文私钥，不符合等保三级、国密安全规范；
服务器一旦被入侵，批量泄露大量企业签署私钥，法律责任巨大；
正规商用 CA 接口不支持外部导入私钥签发证书，仅支持 CA 内部生成密钥模式。
2. 硬件 U 盾（USB-KEY）模式下流程差异
高安全政务 / 金融场景，私钥不在 CA 生成，流程变更为：
实名初审 → 用户激活硬件 KEY → KEY 芯片本地生成公私钥 → 仅公钥上传 CA → CA 签发证书写入 U 盾；
私钥永久锁在硬件芯片，CA 也无法获取。
3. 证书续期时密钥生成逻辑
证书到期前办理续期：
重新复核实名信息 → 核验通过 → 加密机生成全新 SM2 公私钥对 → 签发新证书；
旧的私钥、旧证书标记作废，不再用于签署。
4. 一句话极简流程总结
实名材料提交 → CA 多维度核验身份 → 核验通过触发加密机生成 SM2 公私钥（私钥诞生） → 导出公钥绑定身份信息 → CA 根私钥签名生成公钥数字证书 → 证书交付业务系统，私钥永久托管于 CA 加密机房。

# 5、个人 / 企业 实名认证通过并生成数字证书后，后续 电子签名 / 电子盖章 这一过程进行详细讲解
一、前置基础（已完成环节）
个人 / 企业全链路实名认证通过；
CA 加密机生成一组专属 SM2 公私钥对，私钥加密托管在 CA 密码机，永不对外输出；
CA 使用根私钥对「身份信息 + 用户公钥」签名，生成 X.509 数字证书（仅含公钥、身份、CA 签名）；
业务系统绑定：账号 ↔ 证书 ID ↔ 企业公章 / 个人签名图片模板；
每次签署前必须二次身份校验（短信 / 人脸 / USB KEY PIN），满足《电子签名法》“签名人专有控制” 要求。
核心不变规则：业务系统、前端全程无明文私钥；签名运算全部在 CA 加密机内部完成。
二、完整电子盖章 / 电子签名分步全流程（Node 业务系统 + 第三方 CA 托管密钥架构）
阶段 1：合同文件上传与标准化处理（业务系统层）
操作人员前端上传文件：Word、图片、PDF；
前端二进制流上传至 Node 后端接口；
后端统一处理：
病毒、恶意文件查杀；
Word / 图片转扁平化标准 PDF（防止图层篡改、渲染错乱）；
分配全局唯一 fileId，持久化原始 PDF；
返回 PDF 预览地址、总页数给前端 PDF.js 渲染预览。
阶段 2：前端可视化标记签章位置（仅 UI 交互，无加密逻辑）
用户在预览 PDF 上拖拽印章 / 手写签名；
前端记录每一处签章参数：
目标页码、X/Y 坐标、印章宽高；
签章类型：个人手写签、企业公章、财务章、法人章、骑缝章；
关联证书 ID（当前登录主体对应的数字证书）；
前端将位置数组、fileId、证书 ID 提交后端缓存。
阶段 3：盖章前二次强身份鉴权（法律必备环节）
用户点击【确认盖章】，前端触发二次核验：短信验证码 / 活体人脸 / U 盾 PIN 码；
核验数据传给 Node 服务；
后端双重校验：
① 当前登录账号是否拥有该证书 / 公章的盖章权限；
② 二次核验凭证与实名信息匹配；
校验失败：直接终止流程，返回前端错误；
校验通过：生成一次性短期盖章临时 Token（5 分钟有效期，防止重复调用、越权调用）。
阶段 4：后端发起盖章核心请求，计算文件国密哈希（防篡改根基）
前端携带 fileId、签章位置数组、临时Token、证书ID 调用盖章接口，后端串行执行：
校验临时 Token 合法性、防重放、防重复盖章；
读取完整标准化 PDF 二进制原文；
使用国密 SM3 算法计算全文唯一哈希摘要；
逻辑：文件任意文字、图片、空格改动，SM3 哈希会完全变化，是验签比对核心依据。
阶段 5：调用 CA 接口，在加密机内完成 SM2 数字签名（核心密码运算）
Node 业务系统只传递哈希值与证书 ID，不传递、不持有私钥：
后端请求 CA 开放签名接口，入参：
文件 SM3 哈希值；
用户证书唯一 ID；
业务流水号、时间源请求标识；
CA 服务接收请求后，转发至硬件加密机：
根据证书 ID 匹配内部绑定的企业 / 个人私钥（私钥仅存在加密机安全芯片）；
使用 SM2 私钥对文件哈希做数字签名，生成原始签名值；
同步调用国家合规 TSA 权威时间戳服务器，获取具备法律效力的标准时间戳；
CA 将三样数据返回业务后端：
① SM2 签名值；
② 标准 TSA 时间戳；
③ 完整证书链（用户证书 + CA 中级证书 + 根证书，用于跨设备验签）。
关键重点
私钥全程不出 CA 加密机房，Node 服务内存、数据库不会出现私钥明文；
时间戳不使用服务器本地时间，本地时间可篡改，无司法效力。
阶段 6：PDF 引擎合成可视化印章 + 写入标准 PAdES 签名域（文件落地）
Node 不适合复杂 PDF 底层处理，通常拆分独立 PDF 微服务（Java iText/Go）处理：
6.1 可视化图层渲染（肉眼可见的电子章）
根据之前缓存的坐标、页码、尺寸，叠加透明公章 PNG 图层；
骑缝章特殊处理：将公章图片均等切割为 N 份，每页侧边嵌入片段，视觉形成完整骑缝；
图层层级控制：印章置于文字上层，不遮蔽关键合同条款。
6.2 写入合规数字签名数据（法律效力核心，看不见）
在 PDF 内部创建标准 /Sig 签名字典（PAdES-B-LT 长效签名格式，法院、WPS、Adobe 通用）；
写入全部密码学元数据：
SM2 原始签名值；
TSA 权威时间戳；
用户完整 X.509 证书链；
签署人身份、签署设备 IP、操作人、业务流水；
开启文档锁定：签署区域、全文禁止删除、移动、覆盖印章，普通编辑器无法篡改签名域；
输出全新带签章 PDF 二进制文件，生成全局签署流水号。
阶段 7：异步后置流程（不阻塞前端响应，业务合规必备）
后端开启异步任务并行执行三件事：
区块链 / 第三方存证
将文件 SM3 哈希、签署日志、证书信息、时间戳上链存证；链上数据不可删除、不可篡改，司法可直接调取存证报告。
签约方通知
短信、站内信、邮件推送已盖章合同链接给甲方 / 乙方。
全日志归档
保存原始文件、盖章后 PDF、实名记录、签名元数据、核验记录，合规要求存档不少于 5 年。
阶段 8：盖章结果返回前端展示
后端返回：盖章后 PDF 下载 / 预览地址、签署流水号、在线验签入口；
前端加载带章 PDF 预览，展示签章详情：企业名称、证书编号、权威签署时间；
用户可下载、转发、归档合同文件。
三、独立配套：验签流程（盖章后随时校验真伪）
用户上传已盖章 PDF 至前端；
后端解析 PDF 内置 /Sig 签名域，提取：证书链、SM2 签名值、TSA 时间戳、原始文件哈希；
两步校验：
① 使用证书内的公钥解密签名，还原出签名时的原始 SM3 哈希；
② 重新计算当前 PDF 全文 SM3 哈希，与还原哈希对比；
附加校验：证书是否过期、是否被 CA 吊销、时间戳是否合法有效；
输出验签报告：
哈希一致 + 证书有效：签章真实、文件未篡改，具备法律效力；
哈希不一致：文件后期被修改；证书失效：签名作废。
四、区分「电子签名（个人手写签）」与「电子盖章（企业公章）」流程差异
整体链路完全一致，仅两处区别：
签章模板不同：个人使用手写签名图片，企业使用备案公章 PNG；
绑定证书主体不同：手写签绑定个人数字证书，公章绑定企业数字证书；
底层 SM2 签名、哈希计算、PAdES 写入、存证逻辑无任何区别，同等司法效力。
五、开发层面关键风险约束（贴合你 Node 技术栈）
前端禁止参与任何加密、签名运算
JS 国密库内存易泄露密钥，仅做 PDF 预览、拖拽标记坐标；签名全部交给 CA 加密机。
Node 只做业务调度，剥离 PDF 底层处理
pdf-lib 对国密 PAdES 支持缺陷多、大文件 OOM，单独起 Java/Go PDF 微服务合成签章文件。
私钥永远隔离
业务数据库、Redis、Node 内存不缓存、不接收私钥，仅通过证书 ID 关联签名主体。
二次鉴权不可省略
仅靠登录态直接盖章会违反电子签名法，纠纷时签章直接丧失法律效力。
不依赖本地系统时间
所有签署时间采信 TSA 权威时间戳，否则司法鉴定不认可签署时间。
六、极简串联总结（实名发证完成后完整链路）
上传合同→转标准化 PDF→前端标记盖章位置→二次身份核验→计算文件 SM3 哈希→调用 CA 加密机 SM2 私钥签名 + 获取 TSA 时间戳→PDF 合成公章图层 + 写入标准 PAdES 签名域→异步存证 + 通知归档→前端预览盖章文件→支持随时验签校验真伪。


# 6、盖完电子章后为何通过 wps 就能验证是否合规呢？原理是什么？
WPS 可直接校验电子签章合规性完整原理（结合你前面整套 CA/SM2/PAdES 流程）
一、先回答核心结论
你系统盖章时遵循国标 PAdES-B-LT 国密 PDF 签名规范，并把「SM2 签名值、TSA 权威时间戳、完整 X.509 证书链」全部写入 PDF 标准/Sig签名字典；WPS 内置了符合国家密码局标准的 PDF 签名解析引擎、国密 SM2 验签算法、证书校验模块，能自动读取 PDF 内部加密证据链，完成全套合法性校验，不用依赖你的业务系统。
二、底层完整原理拆解
1. 盖章阶段提前埋入了所有可离线校验的数据（关键）
之前电子盖章流程里，PDF 微服务会把三类不可篡改证据永久嵌入 PDF 文件内部，文件随身携带，不依赖服务端：
1）签署人完整 X.509 数字证书（含公钥、企业 / 个人实名身份、CA 颁发信息）
2）CA 加密机产出的 SM2 数字签名值（用企业私钥对原文哈希加密生成）
3）国家 TSA 权威时间戳（第三方可信时间凭证，自带时间机构签名）
4）PDF 签名锁权限配置、签署原文 SM3 哈希摘要元数据
这就是 WPS 离线验签的基础：所有校验材料都存在 PDF 内部，不用联网你的业务系统。
2. WPS 验签分 4 步核心逻辑（纯本地离线运算）
步骤 1：WPS 读取 PDF 内置标准签名域 /Sig
PDF 是开放标准，所有合规阅读器（WPS、Adobe、福昕）都统一解析/Sig字典，提取：
用户数字证书（提取证书里的SM2 公钥）
SM2 签名密文
TSA 时间戳数据包
签名时原文的 SM3 哈希值 HASH_OLD
步骤 2：WPS 本地重新计算当前整份 PDF 的 SM3 哈希 HASH_NEW
WPS 读取当前打开 PDF 全部文档内容，使用国密 SM3 算法重新计算全文哈希：
如果合同文字、公章位置、页面内容被人为修改，HASH_NEW ≠ HASH_OLD；
只要哈希不一致，WPS 直接标记「文件已被篡改，签章无效」。
步骤 3：使用证书公钥解密 SM2 签名，还原原始哈希
WPS 内置国密 SM2 验签算法；
用 PDF 证书里公开的公钥解密加密签名；
解密成功会得到盖章那一刻原始文件的哈希 HASH_OLD；
对比 HASH_OLD 和步骤 2 算出的 HASH_NEW：
相等 = 签署后文件无改动；
不相等 = 文件后期被编辑、篡改。
核心密码学逻辑：只有对应企业私钥能生成合法签名，配套公钥唯一能解开；如果是伪造公章图片（无底层 SM2 签名），解密会直接失败。
步骤 4：多层合规校验（决定是否判定 “合法有效签章”）
光哈希匹配不够，WPS 还要校验司法合规条件，对应《电子签名法》可靠四要素：
证书身份校验
WPS 内置全国合法 CA 根证书库，校验当前用户证书是否由国家认可 CA 颁发；
证书无根证书信任链 → 提示 “未知发行商，签章不可信”；
读取证书内实名信息（企业统一社会信用代码 / 个人身份证），展示签署主体。
证书状态校验（联网轻校验）
WPS 会根据证书内 CRL/OCSP 地址联网 CA 服务器，检查证书：
是否过期；
是否被企业注销、吊销（经办人离职、公章作废）；
证书吊销则签章直接失效。
TSA 权威时间戳校验
WPS 解析时间戳数据包，用时间戳机构公钥验签时间合法性：
杜绝本地电脑篡改系统时间伪造签署日期；
法院认可时间戳作为签署真实时间证据。
PDF 签名锁校验
检查 PDF 签名域权限配置，确认签署后文档被锁定，禁止编辑修改；若签名锁失效会风险提示。
3. 区分两种 “电子章” 在 WPS 里的完全不同表现
情况 A：你的系统标准 PAdES 国密签章（合规可验）
PDF 内存在标准/Sig签名域，包含证书、SM2 签名、时间戳：
WPS 右键印章 → 查看签名属性：
显示签署人真实企业 / 个人名称；
显示颁发 CA 机构、证书有效期；
显示权威可信签署时间；
提示「签名有效，文档未被篡改」。
情况 B：仅贴透明公章图片（无底层数字签名，不合规）
只是一张图片图层，无/Sig签名字典、无证书、无 SM2 签名：
WPS 识别只是普通图片，不会识别为电子签名，无任何验签入口，司法不认可。
三、为什么 WPS 能做到，普通看图软件不行？
WPS 实现了国标 GB/T 35275 PAdES 国密 PDF 签名完整解析，适配国内电子签章强制要求的 SM2/SM3；
内置全国合规 CA 根证书信任列表，自带国密算法运算库；

四、关键业务意义（对你开发系统的价值）
离线可核验：客户、甲方、法院不需要登录你的业务系统，只用 WPS 打开就能自查合规性，降低举证成本；
司法通用认可：法院、公证处普遍使用 WPS/Adobe 核验电子合同，只要 WPS 显示有效，可直接作为证据；
天然防伪造：仅 PS 一张公章图片无法通过 WPS 验签，缺少底层 SM2 密码学证据链；
全链路自证：PDF 文件自身包含完整证据，不需要依赖你服务端存证记录也能初步验真。
