# 2026-07-14 工作日志

## 国密算法真实现 — SM2/SM3/SM4 接入 libsm crate (已完成)

### 背景
crypto.rs 中 SM2 签名/验签和 SM4 加密/解密是假实现:
- SM2: 摘取 SM3 前 32 字节 + 颠倒后 32 字节拼成 64 字节伪装签名
- SM4: XOR 0xAA 逐字节异或伪装加密
- SM3: 已使用 libsm 真实现 (无需改)

### 改动内容
1. **crypto.rs 重写**:
   - SM2 签名/验签: 使用 `libsm::sm2::signature::SigCtx`
     - 首次调用时由 libsm 随机生成 SM2 密钥对 (sm2p256v1 曲线), 缓存在 `OnceLock`
     - 签名返回 DER 编码 (ASN.1 SEQUENCE { r, s }), 符合 GM/T 0003-2012
     - 验签使用完整椭圆曲线数学验证
   - SM4 加密/解密: 使用 `libsm::sm4::{Cipher, Mode::Cbc}`
     - 真实 SM4-CBC 分组密码, PKCS#7 自动填充/去填充
     - 密钥和 IV 必须各 16 字节, 密文长度为 16 的倍数
   - SM3: 保持不变 (已经是真实现)
   - RSA 签名: 保持假实现 (非国密算法, 仅用于兼容)
   - 新增 `get_sm2_pubkey()` / `get_sm2_seckey()` 获取动态生成的密钥

2. **lib.rs 新增 WASM 导出**: `sm3_hash`, `sm2_sign`, `sm2_verify`, `get_sm2_pubkey`, `sm4_encrypt`, `sm4_decrypt`

3. **签名格式变化**: SM2 签名从 64 字节裸 r||s 改为 DER 编码 (70-72 字节), 符合 GM/T 0003-2012 标准

### libsm API 要点
- `SigCtx::new()` → `new_keypair()` 生成密钥 (uses rand 0.9 via getrandom 0.3 wasm_js)
- `SigCtx::sign(msg, &sk, &pk)` → `Signature`, 用 `der_encode()` 输出 DER
- `SigCtx::load_seckey(32字节)` / `load_pubkey(65字节 0x04前缀)` 反序列化
- `Sm4CipherMode::new(key, Cbc)` → `encrypt(&[], data, iv)` / `decrypt(&[], data, iv)`

## 修复签章嵌入 /Contents 搜索 bug (已完成)

### 问题
盖章后保存/嵌入签名报错: "未找到签名字典 29 的 /Contents"

### 根因
lopdf 0.34 的 `write_dictionary()` 中 `need_separator()` 对 `String` 类型返回 false,
导致 lopdf 输出 `/Contents<000...000>` (无空格), 而代码搜索的是 `/Contents <` (带空格)。
搜索模式不匹配, `find_contents_hex_range` 返回 None。

### 修复
重写 `find_contents_hex_range`:
- 先搜索 `/Contents` 关键字 (9字节)
- 然后跳过可选空白 (空格/换行/回车/Tab)
- 找 `<` hex 字符串起始
- 兼容 `/Contents<` (lopdf 实际) 和 `/Contents <` (旧假设) 两种格式

### 其他 lopdf need_separator 影响
- `/ByteRange[...]` (无空格): 搜索代码先找 `/ByteRange` 再跳空白找 `[`, 已兼容
- `/Type/Sig` `/Filter/Adobe.PPKLite` 等也无空格, 但代码不搜索这些, 无影响
- `replace_byterange_in_bytes` 替换时写入 `/ByteRange [...]` (带空格), 长度微增,
  迭代 ByteRange 计算循环会自动收敛 (2-3 轮)

### 测试脚本
- `tools/test_contents_search.cjs` — JS 端验证两种格式均能正确找到 hex 范围
