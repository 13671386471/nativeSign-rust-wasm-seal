## OFD 渲染空白问题修复

**问题**: sample.ofd 文件打开后渲染为空白，页数显示为1（实际3页）

**根因分析**:
1. `ofd_parser.rs` 的 `parse_text_object()` 函数期望的 XML 格式与实际 OFD 文件不匹配
2. 实际 OFD (GB/T 33190) 使用 `<Size>10.5</Size>` 作为 TextObject 直接子元素表示字体大小
3. 解析器只识别 `<Font Size="10.5">` 格式，导致 font_size 停留在默认值 4.23mm（几乎不可见）
4. 实际使用 `<FontName>SimSun</FontName>` 表示字体名，解析器未处理
5. Canvas 的 `set_font()` 字体大小不受 `ctx.scale()` 变换影响，需要手动乘以缩放系数

**修复内容**:
- **ofd_parser.rs**: 重写 `parse_text_object()`, 增加状态追踪变量 `current_text_elem`, 正确捕获 `<Size>`, `<FontName>`, `<X>`, `<Y>` 等元素的文本内容。同时保持对原有 `<Font Size="">` 属性格式的向后兼容。
- **render.rs**: 修改 `render_ofd_text()` 增加 scale 参数, 字体大小计算改为 `screen_font_size = font_size_mm * scale`, 确保 Canvas 渲染时文字大小正确。

**测试文件**: D:/工作文档/sample.ofd (3页, 每页约31个TextObject, 使用SimSun字体)

## PDF 中文内容空白显示问题修复

**问题**: sample.pdf (STSong-Light CID中文字体, 3页) 在项目中显示为空白

**根因分析**:
1. `index.html` 中 `initialize_pdfium_render()` 的 `local_wasm_module` 参数传了 `window.__pdfiumModule` (PDFium Emscripten 模块) 而不是 wasm-bindgen 导出模块对象
   - 影响: pdfium-render 无法在 PDFium 函数表中查找 `read_block_from_callback_wasm` 等回调函数
2. PDFium WASM 环境下缺少中文字体提供器
   - `use_platform_default_font_provider()` 在 WASM 下始终返回错误
   - PDF 使用 STSong-Light CID 字体 (UniGB-UTF16-H 编码), PDFium 无法渲染

**修复内容**:
- **index.html**: 修复 `local_wasm_module` 参数, 改为 `Object.fromEntries(wasmBindings)` (包含 register_font 等回调函数)
- **新增 font_provider.rs**: 实现中文字体提供器模块
  - `register_font(name, data)` — wasm_bindgen 导出, JS 注入字体数据
  - `ChineseFontProvider` — PdfiumCustomFontProvider 实现, 提供字体数据给 PDFium
  - 支持精确匹配、字符集回退 (ChineseGb2312/ChineseBig5 等)
- **render.rs**: 修改 `get_pdfium()` 在初始化 PDFium 后注册字体提供器
- **index.html**: 添加中文字体加载流程 (本地 → CDN → 备用 CDN)
- **render.rs**: 增加位图诊断日志 (格式、非白色像素数量、空白警告)

**字体来源**: NotoSansSC-Regular.otf (Google 免费中文字体)
- 优先从本地 fonts/ 目录加载
- 回退从 GitHub CDN 加载
