# 2026-07-09 工作记录

## PDF 中文空白 — 字体提供器跨模块修复（续）

### 根因链（最终定位）
1. `pdfium-render 0.9.2` 的 `set_custom_font_provider()` 把 Rust wasm 模块的 extern "C" 函数指针（本模块函数表索引）直接传给 PDFium 的 `FPDF_SetSystemFontInfo`。PDFium 运行在独立 wasm 模块（pdfium.js）中，无法跨模块调用这些指针 → 字体映射时 wasm trap → `panic!()`（wasm_bindings.rs:476，`explicit panic`）。
   - 现象：注册字体提供器后打开 PDF 直接 panic。
   - 修复：`get_pdfium()` 不再调用 `set_custom_font_provider`，改由 JS 端安装（index.html 的 `installChineseFontProvider()`）。

2. JS 端安装提供器时踩的两个坑：
   - 访问 `mod.wasmTable` 会 `Aborted('wasmTable' was not exported...)`。正确路径是 `mod.wasmExports.__indirect_function_table`（或 `mod.asm.__indirect_function_table`）。
   - 回调必须声明与 `FPDF_SYSFONTINFO` 签名一致的形参个数（否则 call_indirect 类型检查 trap）。

3. **当前空白根因（2026-07-09 下午）**：`cbGetFontData` 对 `table != 0` 直接 `return 0`。PDFium 加载 CFF/OTF 字体会请求单个表（如 `CFF `），拿到 0 后拒绝该字体 → 整页空白（non_white_pixels=0）。
   - 修复：`GetFontData` 用 `extractTable()` 从字体字节中按 4 字节 tag 提取对应表返回；`table=0` 返回完整字体。

### sample.pdf 字体特征
- F2 = `/Subtype /Type0` + `/Encoding /UniGB-UCS2-H`（内容流用 Unicode/UCS2 码，非原始 Adobe-GB1 CID）
- 后裔 `/Subtype /CIDFontType0 /BaseFont /STSong-Light`，CIDSystemInfo `Adobe/GB1`
- F1 = Helvetica（嵌入，走不到 MapFont）
- 结论：PDFium 按 Unicode 查字形，Noto Sans SC（带 Unicode cmap）可正确映射；只要提供器被调用且 GetFontData 返回真实表即可渲染。

### 调试手段
- index.html 新增左下角 `#fontDebug` 调试面板（固定定位，半透明黑底）。由 `fontDebug()` 写入，记录：安装状态、MapFont/GetFontData 调用次数与末次参数。避免再靠翻控制台日志猜测。

### 关键文件
- `index.html`：`installChineseFontProvider()`（JS 跨模块字体提供器）、`extractTable()`（sfnt 表提取）、`fontDebug()`、`#fontDebug` 面板
- `src/render.rs`：`get_pdfium()` 不再注册 pdfium-render 的字体提供器

## 续：sample.pdf 仍空白 → 第二个 Bug（cbMapFont 形参顺序错）

### 现象
GetFontData 修复后 sample.pdf 依旧空白（non_white_pixels=0），阳光电源 PDF 正常。

### 新定位：cbMapFont 形参与 FPDF_SYSFONTINFO 签名不符
- 头文件 `pdfium/release/include/fpdf_sysfontinfo.h` 中 MapFont 真实顺序为：
  `(pThis, weight, italic, charset, pitch_family, face, bExact)`（7 参数）
- 但 `index.html` 的 `cbMapFont` 声明为：
  `(pThis, charset, weight, italic, family, face, bExact)`
  → 第 1~4 个位置（weight/italic/charset/pitch_family）被错位接收。
- 后果：`cs = charset|0` 实际拿到的是 **weight(如 400)**，存入 `rec.charset`；
  `cbGetFontCharset` 因此返回 **400（垃圾值）而非 134(GB2312)**。
- 为何 MapFont 仍能返回字体：face 在第 5 位，两边对齐，`readCStr(face)` 正确读到
  "STSong-Light"，被 `/song/` 正则命中 → 仍返回 handle。但报告给 PDFium 的 charset 是错的，
  CID 替换字形时 PDFium 拿不到正确的 GB2312 字符集，导致中文字形无法生成 → 整页空白。
- 连带影响：凡字体名不含 song/hei/kai/cjk/gb/cn/chinese/ming/fang/sc/tc 且本应靠 charset 命中的
  CJK 字体，会同时错过 name 匹配与 charset 匹配 → 必然空白。

### 修复
把 `cbMapFont(pThis, charset, weight, italic, family, face, bExact)` 改为
`cbMapFont(pThis, weight, italic, charset, pitch_family, face, bExact)`，
使 `cs` 真正等于 charset(134)，`GetFontCharset` 返回 134(GB2312)。

### 验证
刷新后看 `#fontDebug`：安装成功 + MapFont>0 + GetFontData>0 无失败 + 控制台无
`[render_page] non_white_pixels=0`，sample.pdf 中文应出现。

## 续：用户反馈仍空白 → 第三次定位（真正的根因）

### 用户提供的调试面板截图确认
- 安装状态: **失败: 无法获取函数表**
- MapFont 调用: 0, GetFontData 调用: 0
- 最近事件: "无法获取 PDFium 函数表, 跳过字体提供器"
- 控制台日志: 全部3页 `non_white_pixels=0`（第2页544/211像素为非白，应为边框线等图形）

### 真正的根因: index.html 第239行 WebAssembly.Table 类型检查错误

**代码**: `if (t && typeof t.length === 'function') { table = t; break; }`

**问题**: `WebAssembly.Table.length` 是 **number** 属性（返回表当前长度），不是 function。
因此即使 `mod.wasmExports.__indirect_function_table` 成功拿到了合法 Table 对象，
`typeof t.length` 返回 `'number'` 而非 `'function'` → 条件 false → 三个 getter 全"失败"
→ 报 "无法获取函数表" → 字体提供器完全未安装 → sample.pdf 中文全空白。

**证据**: pdfium.js 源码确认:
- `__indirect_function_table = wasmTable = wasmExports["__indirect_function_table"]`
- `Module["wasmExports"] = wasmExports`（已导出）
- `assert(typeof wasmExports["__indirect_function_table"]!="undefined")`（断言通过说明确实存在）

**修复1**: 类型判断改为 `t instanceof WebAssembly.Table || (typeof t.length === 'number' && typeof t.get === 'function')`

**修复2**: 同步修正 cbMapFont 形参顺序（之前分析发现的第二个Bug）:
- 改前: `(pThis, charset, weight, italic, family, face, bExact)` — 错误
- 改后: `(pThis, weight, italic, charset, pitch_family, face, bExact)` — 对齐 fpdf_sysfontinfo.h

## 续：第四次 — .length 属性 vs 方法（连锁错误）

### 现象
修复1后报错变成 `异常: table.length is not a function`（函数表这次拿到了）。

### 根因
`WebAssembly.Table.length` 是 **number 属性**，不是方法。代码里有两处
`table.length()` 当函数调用（原扩容逻辑 line 386、覆盖模式扫描 line 403）→ 抛异常。
（我修复1时只改了识别处的判断，漏了后续这两处调用。教训：改 API 用法要全局搜所有调用点。）

### 根本性重构（弃用脆弱的手动函数表操作）
pdfium.js 明确导出了 `Module.addFunction(fn, sig)`（源码 line 82443:
`Module["addFunction"]=addFunction`）。它把 JS 回调编译为**带正确类型签名**的 wasm
包装函数并返回表索引，比"裸 JS 函数塞进 funcref 表 + arity 猜测覆盖"可靠得多，
彻底规避 call_indirect 类型不匹配 trap。

**新逻辑（index.html 第2步）**：
1. 首选 `mod.addFunction(fn, sig)`，签名表 sigMap:
   - Release `vi` / MapFont `iiiiiiii`(ret+7参) / GetFont `iii` / GetFontData `iiiiii`(ret+5参)
   - GetFaceName `iiiii`(ret+4参) / GetFontCharset `iii` / DeleteFont `vii`
2. 回退：`table.grow(n)` + `table.set()`（用 `table.length` 属性，不加括号）
3. 若入口数不足 cbDefs.length 则安装失败并 return

### 关键教训（写入长期约定）
- WebAssembly.Table：`.length` 是属性；`.get(i)`/`.set(i,v)`/`.grow(n)` 是方法。
- 跨模块给 PDFium 装字体回调，优先用 Emscripten 导出的 `Module.addFunction(fn, sig)`。

## 续：第六次 — 根因坐实 + 嵌字体方案验证成功（最终可用修复）

### 结论：JS 字体提供器对 sample.pdf 的 CID 字体机制性失效（死路）
- 多次验证：`MapFont=0` 且 `GetFontData=0`，说明 PDFium 渲染该 CID 字体时**根本不调用系统字体提供器**。
  （注：本项目 index.html 的 `cbGetFont` 被硬编码 `return 0`，但即使修好，实证表明 CID 路径不查提供器。）
- 为什么阳光电源 PDF 正常：其 CJK 字体**已嵌入** PDF，PDFium 直接用嵌入字体，无需提供器 → 正常显示。
- sample.pdf 引用**未嵌入**的 `STSong-Light`(CIDFontType0, Adobe-GB1)，PDFium 拿不到字形 → 整页空白。
- 因此：对"未嵌入 CID 字体的 PDF"，`FPDF_SetSystemFontInfo` 提供器路线在该 WASM 构建下不可行。

### 唯一可靠修复：运行时把 NotoSansSC 嵌入 PDF（已验证）
#### 字体结构事实（scripts/_inspect_font.py 确认）
- `NotoSansSC-Regular.otf` = `OTTO`(CFF), **CID-keyed**, `ROS=(Adobe,Identity,0)`。
- 字形顺序 `cid00001, cid00002, ...` → **CID == GID**（顺序编号）。
- `cmap`: `U+4E2D(中) -> GID cid09544` ⇒ CID 9544。即 Unicode u → 字形名 `cidNNNNN` → CID=N=GID。

#### 之前两次失败的真正原因（同一根因）
- 尝试1 `CIDFontType0C`：直接用内容流里的 **Unicode 码点当 CID**，但 CFF charset 把 CID 映射到 GID，
  故 CID=0x4E2D 落到 GID=20013 的字形 → 错字形/`.notdef` → 豆腐块。
- 尝试2 `CIDFontType2 + FontFile2`：`FontFile2` 要求 **TrueType(glyf)** 字体，却喂了 CFF(OTF) → 子类型不匹配。

#### 正确做法（scripts/embed_cjk_font.py，已落地）
1. 从 cmap 建 `unicode -> cid(GID)` 映射（解析字形名 `cidNNNNN` 取整数）。
2. 重写内容流：/F2 的 UCS-2BE 2字节 Unicode 码点 → 2字节 CID；/F1 的 1字节 WinAnsi → WinAnsi→Unicode→2字节 CID。
3. 字体字典：`Type0`/`Identity-H` + 后代 `CIDFontType0` + `/FontFile3`/`/Subtype`/`CIDFontType0C`，嵌入 OTF 字节。
4. 字体流**只建一份**（`pdf.make_stream` 一次，所有页面 /F1 /F2 引用同一对象）避免重复嵌入。

#### 验证（scripts/verify_glyphs.py，无法看图用"字形形状多样性"判定）
- 投影法把页面墨迹切成字形级小方块，归一化 16x16 哈希统计 distinct 数；豆腐块→distinct≈1~3。
- sample.embedded.pdf：page0=727块/327异形, page1=812/370, page2=1047/422 → **REAL_TEXT**。
- 输出：`D:/workspace/self/rust-wasm-seal/sample.embedded.pdf`（7.2MB，字体嵌入一次）。
- 统计：/F2 字符=3904 全部映射到有效 CID，缺失映射=0。
